# REQUIREMENTS.md - BudgetBook Product Requirements

このファイルは BudgetBook の「何を作るか」「なぜ作るか」を定義する。実装方法は `docs/TECH_SPEC.md`、運用と公開前確認は `docs/SECURITY.md` / `docs/RELEASE_CHECKLIST.md` を正とする。

## 1. プロダクトの目的

BudgetBook は、本人と妻が日々の収入・支出・口座残高を安全に記録し、家計の状態を継続的に把握するための実運用家計簿アプリである。

重視すること:

- 入力が速く、日常運用で負担にならない。
- 口座間移動、繰越、支出比率が会計上正しく扱われる。
- 外出先からも安全にアクセスできる。
- 個人の金融情報を扱うため、セキュリティとデータ保護を最優先する。

## 2. ユーザー

| ユーザー | 説明 | 優先度 |
|---|---|---|
| 本人 | 入力、確認、設定、運用を行う主利用者 | 最高 |
| 妻 | 外出先を含めて家計情報を確認・入力する利用者 | 最高 |
| 管理者 | Django admin、バックアップ、デプロイを扱う運用者 | 最高 |

一般公開ユーザーは存在しない。

## 3. 機能要求

### Must

| ID | 機能 | 受け入れ基準 |
|---|---|---|
| R-001 | ログイン保護 | 未ログインで家計簿画面にアクセスできない |
| R-002 | 取引入力 | 収入・支出を日付、口座、カテゴリ、金額、摘要で登録できる |
| R-003 | 取引編集・削除 | 登録済み取引を修正・削除でき、集計が反映される |
| R-004 | 月次ダッシュボード | 収入、支出、差額、月初繰越、月末残高を確認できる |
| R-005 | 口座残高 | 複数口座の残高が正しく表示される |
| R-006 | 口座間振替 | 口座間移動が収入・支出に混入しない |
| R-007 | 支出構成 | 支出カテゴリ別の構成を確認できる |
| R-008 | 収入比率 | 収入に対する支出割合と残額・超過を確認できる |
| R-009 | カテゴリグループ | 複数カテゴリを分析用グループとして合算できる |
| R-010 | CSV 出力 | 月次の通常取引と振替を CSV で出力できる |
| R-011 | 設定管理 | 口座・カテゴリを追加、編集、有効/無効化できる |
| R-012 | 外部アクセス | Cloudflare Access + Django ログインで本人と妻だけが外出先からアクセスできる |
| R-013 | 月次締め | 対象月の集計スナップショットを保存し、締め済み月の通常取引・振替変更を拒否できる |
| R-014 | 口座残高照合 | 帳簿残高をサーバー側で計算し、実残高との差額を記録できる |
| R-015 | 監査ログ | 重要操作について、操作ユーザー、日時、対象、操作内容を追跡できる |
| R-016 | 確定申告レポート | `tax_tag` でタグ付けされた支出を年単位で集計し、CSV で出力できる（v1.13.0 実装） |
| R-017 | 医療費控除明細管理 | 受診者・医療機関・区分・補填額を構造化保存し、国税庁「医療費控除の明細書」様式の CSV と控除額（差引合計 − min(10 万円, 総所得 × 5%)）を出力できる（v1.16.0 実装） |

### Should

| ID | 機能 | 説明 | 状態 |
|---|---|---|---|
| R-101 | インライン編集 | 取引一覧から文脈を保って編集できる | 実装済（v1.4.0） |
| R-102 | Docker 運用 | Ubuntu Server 上で Docker Compose により安定起動できる | 実装済（v1.5.0） |
| R-103 | 自動バックアップ | SQLite DB と `.env` のバックアップ手順が明確である | 実装済 |
| R-104 | セキュリティチェック | 公開前チェックリストに沿って確認できる | 実装済 |
| R-105 | 生命/地震保険料控除入力 | 控除証明書記載値を入力し、国税庁式（新旧契約・4 枠・合算上限 12 万円・地震 5 万円）で控除額を自動計算できる | 計画中（v1.17.0、仕様書ドラフト済） |
| R-106 | 統合確定申告レポート | 医療費控除 / 生命保険料控除 / 地震保険料控除 を 1 ページで集約し、申告書転記支援テキスト付きで出力できる | 計画中（v1.18.0） |

### Won't

| ID | 機能 | 理由 |
|---|---|---|
| R-901 | 一般公開 | 個人の金融情報を扱うため |
| R-902 | 多人数向け権限管理 | 本人と妻の利用に限定するため |
| R-903 | 外部決済・銀行 API 連携 | セキュリティと運用負荷が高いため |
| R-904 | マイクロサービス化 | 一人運用に不適。Django モノリスを維持 |

## 4. 非機能要求

| 項目 | 基準 |
|---|---|
| セキュリティ | `docs/SECURITY.md` を満たす |
| データ保護 | migration 前にバックアップ手順を明記する |
| 復旧目標 | ローカルバックアップから 30 分以内に復旧できる |
| 保守性 | 仕様外の変更を混ぜず、会計ロジックにはテストを追加する |
| 可用性 | 自宅サーバー再起動後も Docker で自動復旧できる |
| 操作性 | スマホでも主要入力・確認ができる |

## 5. 制約

- 実運用中の SQLite DB を壊さない。
- 秘密情報と実データは Git 管理しない。
- 外部公開は Cloudflare Tunnel / Access を前提とし、ルーターの 80/443 ポート開放は行わない。
- 既存のローカル開発手順を壊さない。
