# v1.7.0 Audit Log

## 目的

本人と妻だけが使う実運用家計簿でも、金融情報の変更は後から追跡できる必要がある。取引、振替、設定、締め、照合の重要操作について、誰が、いつ、何をしたかを残す。

## 仕様

- 重要操作が成功したときだけ監査ログを作成する。
- バリデーションエラー、締め済み月による拒否、削除失敗など、DB に変更が入らない操作は記録しない。
- 監査ログは Django admin から確認する。
- 監査ログの編集・削除 UI は通常運用では使わない。admin では全フィールドを readonly とする。
- 既存 `Transaction`, `Transfer`, `Account`, `Category`, `MonthlyClosing`, `AccountReconciliation` のスキーマは変更しない。

## 対象操作

- 通常取引: 作成、更新、削除、インライン更新。
- 振替: 作成、更新、削除、インライン更新。
- 口座: 作成、更新、有効/無効切替、削除、削除不能時の無効化。
- カテゴリ: 作成、更新、有効/無効切替、削除。
- 月次締め: 作成。
- 口座残高照合: 作成。

## モデル

### AuditLog

- `created_at`: DateTimeField(auto_now_add)
- `user`: User FK nullable
- `action`: create/update/delete/deactivate/close/reconcile
- `target_model`: CharField
- `target_id`: CharField(blank)
- `target_repr`: CharField
- `summary`: CharField(blank)
- `metadata`: JSONField(default=dict, blank=True)

## 受け入れ条件

- 取引作成で監査ログが 1 件作成される。
- 取引更新で監査ログが 1 件作成される。
- 取引削除で削除対象 ID を含む監査ログが残る。
- 月次締め作成で監査ログが作成される。
- 口座残高照合作成で監査ログが作成される。
- バリデーションエラーでは監査ログが作成されない。
- 既存テーブルへの破壊的 migration がない。
- `python manage.py test ledger` が通る。

## 非対象

- 専用の監査ログ閲覧画面。
- 監査ログの外部転送。
- 監査ログの自動削除。
- 過去操作の backfill。

## ロールバック

- `migrate ledger 0005` へ戻すと監査ログテーブルは削除される。
- 既存の取引・振替・口座・カテゴリ・締め・照合データには影響しない。
- ロールバックすると監査ログは失われるため、必要なら DB バックアップから保全する。
