# v1.8.0 CSV Import

## 目的

旧家計簿アプリや銀行明細から取引を取り込めるようにし、移行・明細精査の手間を下げる。`v1.0` 以降の CSV エクスポート（`/transactions/export/`）と対称な入口を用意する。

## 仕様

### 入力ファイル

- 受け付けるフォーマットは `transaction_export` の出力と同じヘッダを基本形にする。
  - ヘッダ: `日付, 種別, 口座, カテゴリ, 金額, 摘要, メモ`
  - `種別` は `収入` / `支出` のみ。`振替` 行は v1.8.0 では非対象（明細プレビューでは「スキップ」として表示する）。
- エンコーディングは UTF-8 (BOM 付き / なし) と Shift_JIS を自動判定する。
  - 判定は `chardet` を使わず、まず UTF-8 でデコードし、`UnicodeDecodeError` が出た場合に Shift_JIS にフォールバックする方式。両方失敗した場合はファイル全体を拒否する。
- 改行は LF / CRLF どちらも許容する。
- 1 ファイルの最大行数は **1000 行**、最大バイト数は **1 MiB** とする。超過時はアップロード自体を拒否する（DoS とメモリ消費の上限）。

### フロー（2 段階）

1. **アップロード**: `/transactions/import/` で `multipart/form-data` POST。サーバ側で行をパースし、全行をセッションではなく一時的に画面に埋め込み（hidden input、または再 POST 可能なテキストエリア）プレビュー画面を返す。
2. **確定**: プレビュー画面で「取り込む」ボタンを押すと、選択された有効行のみ `Transaction.objects.bulk_create` で一括登録する。確定処理は `transaction.atomic` でラップする。

セッションに乗せない理由: SQLite + 個人運用前提で、CSV データはユーザー入力の塊。ハイドレートし直しに必要な情報だけを HTML form の hidden で持ち回し、サーバ側で再 validation する。

### 検証ルール（プレビュー時）

各行を以下の順で評価し、`status` を付ける:

- `ok`: 全フィールドが妥当。確定対象。
- `warning_duplicate`: 同日・同口座・同カテゴリ・同額の `Transaction` が既に DB に存在する場合。チェックボックスでデフォルト ON（取込）。
- `error_*`: 取込不可。チェックボックスは無効化。
  - `error_date`: 日付パース失敗 / 未来日付（`clamp_future_month` の翌月以降）
  - `error_amount`: 1 以上の整数でない
  - `error_account`: 名前で照合できない（既存口座のみ許容、自動作成しない）
  - `error_category`: 名前で照合できない、または `種別` (収入/支出) が `Category.kind` と不一致
  - `error_closed_month`: その月が `MonthlyClosing` で締め済み

口座・カテゴリは **名前完全一致**で照合する。大文字小文字・前後空白は正規化（strip）するが、表記揺れは自動補正しない（ユーザに気づかせる）。

### CSV インジェクション対策

プレビュー画面で `description`, `memo`, `category.name`, `account.name` をテンプレ表示する際は Django autoescape に任せる。`description` 先頭が `=`, `+`, `-`, `@`, `\t`, `\r` の場合は先頭に `'` を付けて警告フラグ `csv_unsafe` を立てる（取込自体は許容、表示で目立たせる）。

### セキュリティ・データ保護

- `LoginRequiredMixin` 相当のログイン必須。
- CSRF はフォーム POST に必須。
- アップロードサイズは `request.FILES` のサイズで先行チェック。
- ファイル拡張子は `.csv` のみ許容（MIME は信用しない）。
- 確定処理時に `record_audit(action='transaction_import', target=...)` で件数・口座別件数・ファイル名を残す。
- 全行 error の場合は確定ボタンを表示しない。

### 画面

- `/transactions/import/`
  - GET: 空のアップロードフォーム + 「対応フォーマット」説明 + 「テンプレート CSV ダウンロード」リンク（既存 export を流用、空月でも OK）
  - POST (multipart, `csv_file` フィールドあり): プレビュー表示
  - POST (`confirm=1` + `rows[]` の hidden 群): 確定実行 → ダッシュボードへリダイレクト（`messages.success` で「N 件取り込みました」）
- 既存ダッシュボードの「エクスポート」ボタン横に「インポート」リンクを追加。

### モデル変更

- なし。`Transaction` も `AuditLog` も既存のままで足りる。
- migration なし。

## 非対象

- 振替（`Transfer`）の取込
- 口座・カテゴリの自動作成
- 銀行 CSV など独自フォーマットからの自動マッピング（将来は `formatters/` で拡張可能にだけする）
- スケジュール取込・cron
- バックグラウンドジョブ化（同期 POST で完結させる）

## テスト観点

`budgetbook/ledger/tests/test_csv_import.py` を新規追加。

- UTF-8（BOM 付き / なし）と Shift_JIS の自動判定
- 1000 行超でアップロード拒否
- 1 MiB 超でアップロード拒否
- ヘッダ不一致でエラー
- 日付・金額・口座名・カテゴリ名の個別エラー検知
- 種別とカテゴリ kind の不一致検知
- 月次締め済み月への取込拒否
- 重複検出（同日同口座同カテゴリ同額）
- CSV インジェクション疑いの行に `csv_unsafe` フラグ
- 確定後の `Transaction` 件数と `AuditLog` レコード作成
- 確定時に `atomic` で部分失敗が全件ロールバックされること
- 振替行はスキップされ、`Transfer` が作成されないこと
- 全行 error の場合の確定ボタン非表示

## セキュリティ影響

- 入力経路追加。サイズ上限・行数上限・encoding 限定・MIME 非依存判定で抑え込む。
- `record_audit` で取込操作を残す。

## DB 影響

- スキーマ変更なし。`bulk_create` による INSERT のみ。FK は事前 validation 済みなので IntegrityError は理論上発生しないが、`atomic` で保護する。

## 既存データへの影響

- 既存取引は読まれるだけ（重複検出のため）で更新・削除されない。

## ロールバック方針

- 機能をデプロイから外しても既存 DB に影響なし（モデル変更なし）。
- 万一バグで誤った取込が走った場合: `AuditLog` の `transaction_import` レコードに作成 ID リストを残しておけば、リスト指定で `Transaction.objects.filter(id__in=...).delete()` で巻き戻せる。本実装でも `metadata={'created_ids': [...]}` を残す。

## 受け入れ条件

- `/transactions/import/` GET 200、ログイン未済は 302。
- 正しい CSV のアップロードでプレビューが表示される。
- 確定後、ダッシュボードに N 件が反映される。
- 締め済み月の行は取込されない。
- 全テスト (現行 288 + 新規) が緑。
