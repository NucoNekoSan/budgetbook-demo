# v1.11.0 LoanProfile 利息自動計上 (management command)

## 目的

リボ払い / 有利子負債口座（`LoanProfile.annual_rate_bp > 0`）について、当月利息相当額を **支出 Transaction として自動生成** する。手動で毎月「金利・手数料 ¥5,445 をクレジットカードA に計上」するのを省略し、複利累積を会計データに反映する。

ユーザの実状（2026-05-01 月初時点）:

| 口座 | 残債 | 年利 | 月利息 (round) |
|---|---|---|---|
| クレジットカードA | ¥388,897 | 16.80% | ¥5,445 |
| クレジットカードB | ¥99,436 | 15.00% | ¥1,243 |
| クレジットカードC | ¥298,171 | 18.00% | ¥4,473 |
| ショッピング分割 | ¥34,300 | 0% | — (対象外) |
| 分割返済B | ¥184,907 | 0% | — (対象外) |

毎月手動入力していた 3 件分（合計 ¥11,161/月）を 1 コマンドで生成する。

## スコープ

### 対象

- `python manage.py accrue_loan_interest --month YYYY-MM` を新規追加
- `annual_rate_bp > 0` の LoanProfile のみ対象
- 月末日付の **支出 Transaction** を 1 件 / LoanProfile で生成
- `--dry-run` (デフォルト) と `--apply` の二択
- 既に同月同口座の利息 Transaction が存在する場合は **二重計上を拒否**
- AuditLog に `action=create`, `target_model=Transaction`, `metadata={"source": "accrue_loan_interest", "month": "..."}` を記録
- `--apply` は `transaction.atomic` 内で実行

### 非対象（このバージョンでは扱わない）

- **元金返済 Transfer の自動生成** — `LoanProfile` に「引落元の資産口座」フィールドがないため、別バージョン (v1.12.0 候補) で扱う
- 日割計算 — 月末まとめ計上のみ
- 年単位の総括レポート — 確定申告レポート (別バージョン候補) で扱う
- UI からの実行 — CLI のみ。Web UI は将来検討
- 既存利息 Transaction の更新・削除

## 計算ロジック

- 月利率 = `annual_rate_bp / 10000 / 12`
- 元本 = 当該月初時点 (`YYYY-MM-01`) の **負債口座残高の絶対値**（`all_account_balances` を流用）
- 利息額 (整数円) = `round(元本 × 月利率)` （半端は四捨五入、後述の理由）
- 利息額が **0 円以下なら生成しない**
- 計上日 = 対象月の月末日付
- カテゴリ = 設定ファイルで指定する「金利・手数料」カテゴリ（後述）

### 端数処理の理由

Django の `IntegerField` を維持し、整数円で扱う既存方針と一致させる。クレジットカードの利息明細も実際には円未満は四捨五入されるケースが多いため、`round()` でユーザ感覚と合わせる。

## 設定

`settings.py` に以下を追加:

```python
# v1.11.0: 利息自動計上で使うカテゴリ名（完全一致）
LOAN_INTEREST_CATEGORY_NAME = os.getenv('LOAN_INTEREST_CATEGORY_NAME', '金利・手数料')
```

- 名前 (`Category.name`) で完全一致検索
- 該当カテゴリが存在しない、または `kind != 'expense'` の場合は **コマンドを即エラー終了**
- 該当カテゴリが `is_active=False` でも生成可（記帳目的のため）

## 月次締めとの相互作用

- 対象月が **既に MonthlyClosing で締められている場合は拒否**（`--apply` も `--dry-run` 表示も含めてエラー）
- 締めの取消 (v1.6.0 の機能) で再オープン後は再実行可
- `--dry-run` は締めの有無に関わらず計算結果のみ表示

## CLI 仕様

```bash
# 当月（システム日付の月）を dry-run
python manage.py accrue_loan_interest

# 月指定で dry-run
python manage.py accrue_loan_interest --month 2026-05

# 確定実行
python manage.py accrue_loan_interest --month 2026-05 --apply

# 特定の LoanProfile のみ
python manage.py accrue_loan_interest --month 2026-05 --account "クレジットカードA" --apply
```

出力例 (`--dry-run`):

```
[DRY-RUN] 2026-05 利息計上 (3 件)
  クレジットカードA  元本 ¥388,897  月利 1.4000%  利息 ¥5,445
  クレジットカードB  元本 ¥99,436   月利 1.2500%  利息 ¥1,243
  クレジットカードC  元本 ¥298,171  月利 1.5000%  利息 ¥4,473
合計利息: ¥11,161
※ --apply で確定します。
```

出力例 (`--apply`):

```
[APPLY] 2026-05 利息計上 (3 件)
  クレジットカードA  利息 ¥5,445  Transaction#421 作成
  クレジットカードB  利息 ¥1,243  Transaction#422 作成
  クレジットカードC  利息 ¥4,473  Transaction#423 作成
合計利息: ¥11,163
AuditLog 3 件記録。
```

## セキュリティ・データ保護

| 項目 | 内容 |
|---|---|
| セキュリティ影響 | CLI のみ。Web 公開面なし。新規エンドポイントなし |
| DB 影響 | Transaction を新規 INSERT のみ。既存行は変更しない |
| 既存データへの影響 | なし。新規行のみ生成 |
| destructive migration | **なし**（新規モデル・新規カラムなし。コード追加のみ） |
| バックアップ | 利息計上自体が金額影響を持つため、`--apply` 前に通常通り `db.sqlite3` の手動コピーを推奨。仕様書のセットアップ節に明記 |
| ロールバック | 失敗時は `transaction.atomic` でロールバック。`--apply` 後に取り消したい場合は AuditLog で対象 Transaction を特定し、Web UI から削除（既存機能） |

## ロールバック方針

- 機能フラグなし。新規コマンドの追加のみのため、git revert で完全ロールバック可能
- `--apply` で生成済みの Transaction は AuditLog から特定し、Web UI から削除すれば原状復帰
- `transaction.atomic` により部分失敗時の中途半端な状態は発生しない
