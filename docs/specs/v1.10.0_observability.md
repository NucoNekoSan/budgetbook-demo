# v1.10.0 観測性強化

## 目的

運用中の BudgetBook の状態を可視化し、異常をユーザー自身が早く検知できるようにする。具体的には:

- 主要メトリクス（取引件数、口座数、AuditLog 件数など）の JSON エンドポイント
- ログイン履歴の閲覧 UI（django-axes が既に記録している `AccessLog` を表示）
- 5xx エラー発生時の任意 Webhook 通知（Discord / Slack / メール のいずれか）

## 仕様

### A. `/metrics` JSON エンドポイント

- パス: `/metrics`
- 認証: **ログイン必須**。個人運用 + Cloudflare Access 前段でカバーされるが、生 URL を晒さない方針と整合させる。
- 形式: `application/json; charset=utf-8`
- キャッシュ: `Cache-Control: no-store`
- 返却内容:

```json
{
  "version": "1.10.0",
  "generated_at": "2026-05-12T11:30:00+09:00",
  "counts": {
    "accounts": {"total": 9, "asset": 4, "liability": 5, "active": 9},
    "categories": {"total": 23, "expense": 18, "income": 5, "active": 23},
    "transactions": {"total": 54, "this_month": 7},
    "transfers": {"total": 3, "this_month": 0},
    "monthly_closings": 5,
    "reconciliations": 3,
    "audit_logs": {"total": 42, "last_7_days": 12}
  },
  "balances": {
    "asset_total": 1000000,
    "liability_total": -1500000,
    "net_worth": -500000
  },
  "axes": {
    "recent_failed_logins_24h": 0,
    "currently_locked_usernames": 0
  }
}
```

- 取引明細などの **生データは含まない**（観測用途に絞る）。
- 残高は資産合計・負債合計・正味資産の 3 値のみ。

### B. ログイン履歴ビュー

- パス: `/settings/login-history/`
- 認証: ログイン必須。
- 元データ: `axes.AccessLog`（成功した access）+ `axes.AccessAttempt`（失敗）
- 表示項目: 日時、種別（成功 / 失敗）、ユーザー名、IP、User-Agent（短縮）、Path
- ページング: 50 件 / page、最大 30 日分
- 並び順: 新しい順
- 削除機能なし（監査目的）

### C. エラーメール通知

- トリガー: HTTP 5xx の例外発生時（4xx は対象外）
- 設定:
  - 通知先: 環境変数 `ERROR_NOTIFY_TO`（カンマ区切り複数 OK、未設定なら無効）
  - 送信元: `DEFAULT_FROM_EMAIL`（既存 / なければ `budgetbook@localhost`）
  - SMTP: 標準の Django EMAIL_* 環境変数（`EMAIL_BACKEND` / `EMAIL_HOST` / `EMAIL_PORT` / `EMAIL_HOST_USER` / `EMAIL_HOST_PASSWORD` / `EMAIL_USE_TLS`）。未設定時は `console` backend にフォールバック（ローカル開発でログに出るだけ）。
- 実装: Django logging の `ERROR` レベル以上に対する custom handler
- 件名: `[BudgetBook] 5xx <ExceptionClass> at <path>`
- 本文（プレーンテキスト）:
  ```
  時刻: 2026-05-12T11:30:00+09:00
  ホスト: budgetbook-prod
  メソッド: POST
  パス: /transactions/new/
  ステータス: 500
  例外: KeyError: 'account'
  トレースバック（先頭 5 行）:
    File "...", line 12, in foo
    File "...", line 34, in bar
    ...
  ```
- **含めないもの**:
  - リクエストボディ・cookie・session 値
  - 取引データ / 個人情報
  - 完全なトレースバック（攻撃の手がかりにならないよう先頭 5 行に切る）
- レート制限: 同じ `(path, exception_class)` の組は **5 分間に 1 回** までしか送信しない（バースト抑止）。プロセス内のメモリで管理（再起動でリセットされる、それで OK）。
- 送信失敗時: `logger.warning` でローカルログに残すだけ。アプリの動作を妨げない。
- 送信は別スレッド（`threading.Thread(daemon=True)`）でファイア & フォーゲット。

## モデル変更

- なし。`axes.AccessLog` / `axes.AccessAttempt` は既に存在。
- migration なし。

## 非対象

- Prometheus エクスポーター形式
- グラフ表示（数値テーブルのみ）
- セッション単位のリクエストログ
- Webhook 送信（メールのみ）
- 通知の retry / queue（ベストエフォート）
- ログイン履歴の手動削除

## 受け入れ条件

- `/metrics` GET 200 / JSON、ログイン未済なら 302
- `/metrics` のレスポンスに上記キーが揃っている
- `/settings/login-history/` GET 200、最新の成功・失敗ログが表示される
- `ERROR_NOTIFY_TO` 設定時のみメール handler が attach される
- `ERROR_NOTIFY_TO` 未設定で起動エラーが出ない
- 既存 331 件 + 新規テストが緑

## テスト観点

`budgetbook/ledger/tests/test_observability.py` 新規追加。

### /metrics
- ログイン必須（未ログインは 302）
- 200 + JSON、必須キーの存在
- 残高計算が `balance.py` と一致
- 取引件数が DB と一致

### ログイン履歴
- ログイン必須
- 表示されるエントリに成功・失敗が含まれる
- 30 日より古いものは表示されない（クエリ条件）

### エラーメール
- `ERROR_NOTIFY_TO` 未設定で handler が attach されない
- 設定時に attach され、5xx 発生で件名・本文が組み立てられる
  （実際の送信は `django.core.mail.outbox` を使った Django のテストメール backend で検証）
- 本文にリクエストボディ・cookie が含まれない
- 同じ exception で 2 回目以降は送信されない（5 分以内）

## セキュリティ影響

- `/metrics` は集計値のみ。生取引データを返さない。ログイン必須。
- メール送信は **outbound SMTP** が発生する。Gmail SMTP を使う場合はアプリパスワード必須。`.env` のサンプルに `# ERROR_NOTIFY_TO=` をコメントアウト状態で記載。
- メール本文に PII / 取引情報 / cookie を含めない（テストで検証）。
- ログイン履歴は IP / User-Agent を保持する。これは axes が既に持っているデータの表示のみで、新規記録は増やさない。

## DB 影響

- なし。

## ロールバック方針

- `/metrics`, `/settings/login-history/` の URL を外せば終了。
- メール handler は環境変数 `ERROR_NOTIFY_TO` を空にすればロード時 attach されない（コード変更不要）。
